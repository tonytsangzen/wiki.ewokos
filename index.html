Design Philosophy

Why a Microkernel?

In a microkernel architecture, only the absolute minimum runs in privileged kernel mode: scheduling, IPC, and memory management. Everything else - device drivers, file systems, network stacks - runs as isolated user-space processes. A fault in a driver cannot crash the kernel. The system stays resilient.

Fork + Copy-on-Write

Child shares read-only pages with parent via map_page_ref. Physical page copy deferred until first write, saving memory and startup time.

Capability-Based Privilege

Root processes (uid=-1) bypass syscall checks. User processes (uid>0) restricted from DMA, MMIO mapping, IRQ registration, and priority changes.

Signal Delivery

Two signals: SYS_SIG_STOP (suspend) and SYS_SIG_KILL (terminate). Delivery hijacks the target's context to run its user-space signal handler.

Inter-Process Communication

Synchronous Message Passing

IPC is the backbone of every microkernel. EwokOS uses synchronous, rendezvous-style message passing: the kernel mediates but never buffers. Data flows directly between client and server address spaces.

1 Client issues call SYS_IPC_CALL
2 Kernel blocks client, wakes server ipc_do_task()
3 Server reads caller arguments SYS_IPC_GET_ARG
4 Server writes return data SYS_IPC_END
5 Client retrieves response SYS_IPC_GET_RETURN

Service discovery via the core name server using string identifiers like IPC_SERV_VFS. Supports IPC_NON_RETURN for fire-and-forget calls and IPC_LAZY for deferred processing to optimize throughput.

Copy-on-Write Fork

Child shares read-only pages with parent. Physical copy deferred until first write via map_page_ref reference counting.

Physical Page Allocator

Manages 4K blocks from allocable physical memory. Buddy-style coalescing reduces fragmentation in the free page pool.

Shared Memory

Up to 128 MB of shared memory mapped into multiple process address spaces for zero-copy data exchange between services.

Memory Addressing

Page Ref Counting

Each physical page tracks how many virtual mappings reference it via map_page_ref. Page freed only when count reaches zero.

TLB Coherence

Page table modifications trigger architecture-specific TLB invalidations. ARM uses TLBI instructions; x86 uses INVLPG. SMP cores receive TLB shootdown IPIs.

Kernel/User Split

Higher half kernel design: top VA bits select kernel (TTBR1) vs user (TTBR0) translation tables. Kernel mapped identically across all process address spaces.

Why EwokOS

Microkernel Advantages in Practice

◈

Fault Isolation

A crash in a device driver or file system server cannot bring down the kernel. Each service is an isolated process with its own address space and privilege level.

▥

Modular by Design

Add or remove system services at runtime. Start only the drivers your hardware needs. The system adapts to the machine, not the other way around.

◒

Fine-Grained Security

Each service runs under its own UID/GID with capability-based syscall access. Root privilege is not required for driver operation.

⟳

Live Service Restart

Failed services restart without a full reboot. The core name server re-registers services and clients reconnect transparently via fresh IPC lookups.

⬒

SMP Scalability

Per-core scheduling eliminates global lock contention. Services naturally distribute across available cores via the scheduler's load-aware dispatch.

◫

Educational Value

Clean architecture with well-defined boundaries makes EwokOS an ideal platform for learning OS internals. The codebase is approachable, with inline race-condition analysis.

Arch-Specific Boot

Each platform has its own boot.S: ARM sets up exception levels and GIC, x86 enters long mode, RISC-V initializes SBI. Platform code lives in kernel/platform/.

SMP Secondary Cores

Primary core initializes kernel, then wakes secondaries via PSCI (ARM) or APIC IPI (x86). Each core gets its own idle process and run queue.

Init Process Chain

Kernel loads VFS daemon from ext2 rootfs. VFS then spawns core name server, device drivers, and session manager via init.rd scripts.

Get Started

Build. Run. Explore.

Clone the repository and boot EwokOS in QEMU in under a minute.

# Clone the repository
$ git clone https://github.com/misazhu/EwokOS
$ cd EwokOS/machine.virt
# Build and run (ARM64 QEMU virt)
$ make run
# SMP mode with 4 cores
$ make run SMP=yes
# Full build with extra apps + X Window
$ make run SMP=yes EXTRA=yes
